# Decision Log

Resolves the ambiguities raised against `b2b-app-requirements.md` v0.1. Each
entry is a working decision; revisit only when the listed trigger fires.

| # | Topic | Decision | Trigger to revisit |
|---|---|---|---|
| 1 | Repo shape | pnpm monorepo scaffolded by Shopify CLI. `apps/worker` (Hono on Workers, hosts API + App Proxy), `apps/admin` (Polaris + App Bridge 4 + Remix), `extensions/theme-app-extension`, `extensions/functions/{cart-transform,cart-validation,delivery-customization}`, `packages/shared` for cross-target pricing logic. | Worker bundle size > 1MB compressed, or build time > 90s. |
| 2 | Image processing | Cloudflare Images for variants (original/1200/300). Simpler API, native variants, predictable cost. | Monthly image bill exceeds 2× equivalent R2 + Transforms cost at projected volume. |
| 3 | R2 bucket layout | Single shared bucket, keys prefixed `shops/<shop_id>/<entity>/<id>/<variant>`. | A tenant contractually requires physical isolation, or R2 list operations become a bottleneck. |
| 4 | D1 partitioning | Single logical DB with `shop_id` column; no per-shop DBs at launch. | Any single shop exceeds 5M rows in `asset_downloads`, or query p95 > 50ms. |
| 5 | Catalogs ↔ Markets ↔ Tiers | Catalog assigned to the Company Location provides base price. Markets contributes currency/locale/tax only. Our `cart-transform` Function discounts off whatever line price Shopify passes in. Requirements doc §4.3 to be amended. | Shopify changes the Catalog-Location binding model. |
| 6 | Storefront gating | Liquid server-rendered hide of B2B-only PDPs (returns 404 in theme template) when `customer.b2b?` is false. Collections rely on Search & Discovery metafield filter excluding `b2b.b2b_only == true`. Theme App Embed Block performs client-side tier-price refinement only. Drop the "redirect to alternate collection" idea entirely. | Shopify ships a first-class B2B-only product flag. |
| 7 | Buyer auth | Customer Account API magic-link login only. No password-set flow. §7 Step 4 to be amended. | Shopify deprecates magic-link in Customer Accounts. |
| 8 | Application document uploads | Worker issues signed R2 PUT URLs; browser uploads directly to R2; form POST contains R2 keys + metadata only. Avoids App Proxy body-size and timeout limits. | Cloudflare changes R2 signed-PUT semantics. |
| 9 | App Proxy prefix | Read from `shops.settings_json.app_proxy.{prefix,subpath}`. No hardcoded `/apps/b2b-companion/...` paths in code or copy. | Shopify removes per-app proxy prefix config. |
| 10 | Tier cache in localStorage | UX cache only — used for instant price display on subsequent page loads. Functions and any server route that gates access **must** re-resolve from Company metafield / D1 on every request. Documented in the storefront block module header. | n/a — invariant. |
| 11 | Webhook list | Add `app/scopes_update`. Defer `app_subscriptions/update` until Phase 5 billing wiring. | App Store review flags additional required topics. |
| 12 | Pilot merchant constants | NZ-based merchant on Shopify Advanced, Dawn-based theme, ~20 customers tagged `wholesale`. Tax-ID validators prioritise NZ IRD/GST format; AU ABN deferred. | Pilot merchant differs materially, or wholesale customer count grows past ~100 before launch. |

| 17 | Internal operator console | A small `/_ops/*` route on the same Worker, gated by **Cloudflare Access (Zero Trust)** with SSO. Used by the app vendor's own staff for webhook replay, queue retries, cross-tenant reports, GDPR audits, key rotation, per-shop feature flags. Not exposed to merchants or buyers; not Shopify-authenticated. All actions audit-logged with the operator's SSO identity. | Team grows past ~5 operators (move to per-role IAM), or a customer demands physical separation. |
| 18 | Buyer-facing dealer portal surface | **Worker-rendered HTML page served via Shopify App Proxy at `<shop>/apps/<subpath>/portal`.** Replaces the earlier `customer-account.page.render` UI extension, which Shopify's validator kept rejecting with "the target customer-account.page.render cannot be combined with any other targets" even when the toml had a single target. Auth = App Proxy HMAC on every hit (entry, /api/*, /static/*); no session cookie. A tiny `customer-account.order-index.block.render` extension (`b2b-portal-link`) surfaces the link from inside the customer account chrome. Trade-off: buyers leave the customer-account chrome when they click through, gaining a fully controlled UX and deep-link support. | Shopify's customer-account UI extensions framework stabilises around the page-render target and a sibling-block link pattern; or merchants report friction with the cross-domain transition. |
| 19 | GDPR webhook processing model | **Queue + 7-day stand-down + daily-cron sweep.** Receive handlers (`handlers/gdpr-*.ts`) insert one row into `gdpr_requests` with `due_at = received_at + grace`. The existing daily cron (`0 9 * * *`) runs `runGdprSweep` alongside `runActivationNudgesScan`; the sweep claims due rows and dispatches: `customer_data_request` → email export via Resend, `customer_redact` → `redactCustomer`, `shop_redact` / `app_uninstall_purge` → `redactShop`. Grace windows: 1h for data exports, **7 days** for `customer_redact` and `shop/redact`, 30 days for the uninstall purge (Shopify's mandatory minimum). Stand-down rationale: Shopify already delays `customers/redact` by 10 days and `shop/redact` by 48 h, so our additional 7 days lands well inside the 30-day legal deadline while protecting against accidental requests. Admin UI (`apps/admin/app/routes/gdpr.tsx` + `routes/admin-gdpr.ts`) lets staff Cancel or Process Now during the window. | Shopify shortens its native pre-delivery delays, or a merchant compliance audit flags the additional latency. |
| 20 | Outbound `fetch` origins | Hosts/origins for any server-side `fetch` (Worker base URL, third-party APIs) must come from `context.cloudflare.env` (or other server-only config), never from request input. Only path segments and bearer/session tokens may be taken from the request, and only when the origin is fixed server-side. The `dismiss-plus-banner` SSRF (issue #30) is the regression case this codifies. | A feature legitimately needs to fetch a buyer-supplied URL — then route it through an explicit allowlist helper (private-IP block, scheme whitelist, host pattern) rather than relaxing this rule. |
| 21 | Site-wide tier-price display | Tier-price display is a single app-config toggle (`priceDisplay.siteWide` in `AdminSettings`) mirrored to Shop metafield `b2b.price_display`; the existing `b2b-price` App Embed reads it and, when on, runs on **every template** (the PDP-only `template contains 'product'` guard is lifted) instead of PDP-only. The client engine **overlays our tier delta on top of whatever price Shopify already rendered** (the native catalog price), parsing the displayed money (currency symbol + grouping preserved). Both percent **and** fixed-amount tiers are computed client-side: the `cart-transform` Function discounts **every** line uniformly with **no per-product exclusions** (verified against `extensions/functions/cart-transform/src/index.ts`), so the same per-unit op on the displayed price is parity-exact with checkout. That makes a server-side per-variant `/tier-prices` map unnecessary for parity — it was in the original v2 sketch but is **dropped** (it would add a network hop without changing the result while discounting stays uniform). Parity safety instead rests on two guards: (a) the overlay only ever reduces a price (skips when `new >= base`), and (b) tier resolution is **Plus-aware** — `/tier-context` returns `tier: null` on Plus shops (where the Function early-returns), so neither the PDP block nor the site-wide overlay shows a discount checkout won't honour. B2B membership is still resolved on Plus so `b2b_only` gating keeps working. A `MutationObserver` re-runs the scan for AJAX-rendered surfaces (cart drawer, quick view, infinite scroll); it's idempotent (skips `data-b2b-overlaid` nodes) so it can't loop. Product cards live inside theme section loops, **not** app-block slots, so per-card Liquid injection isn't possible theme-agnostically — the DOM-scan overlay is the portable path. Assumes Online Store → Preferences → Restrict store access → require login (B2B force-login): the overlay is still safe without it (no resolved tier → no-op → public price), but force-login is the supported configuration. | The `cart-transform` Function gains per-product exclusions or per-variant base divergence (then a server `/tier-prices` map becomes necessary again), or Shopify exposes product-card app-block slots / native storefront rendering of Function output. |
